Note: April 2023 didn’t include a human‑authored code change in the repository. This article captures a topic that kept
coming up with users: how to bring diagrams‑as‑code into a Word‑centric workflow without fragile manual steps.

== Why this matters

Big companies love Word for final deliverables: proposals, audit reports, architecture overviews, and executive decks.
They also love traceability and repeatability. That’s where diagrams‑as‑code (DaaC) shines: instead of hand‑drawing
boxes that drift, you generate diagrams from text sources under version control, then include them in `.docx`
documents. As a solo maintainer, I gravitate toward techniques that scale my time. Automating diagram generation reduces
support questions and eliminates a whole class of “could you update this picture?” requests.

== The practical setup

- Choose a generator you can run headless in CI: PlantUML, Mermaid CLI, Graphviz/Dot, or Kroki (HTTP gateway that
  supports many formats). Favor tools available in your enterprise CI image or easy to vend via container.
- Keep diagram sources beside the text that references them. For example, store PlantUML `.puml` next to the AsciiDoc or
  test that explains the component, and output `.png`/`.svg` artifacts into a known `assets/` folder.
- Make generation part of the build. In Maven, add an execution phase that runs your diagram generator, writes assets,
  and then stamps the `.docx` template with Office‑stamper. In CI, cache downloads and Docker layers to keep feedback
  loops fast.
- Wire placeholders for images. In your Word template, add a placeholder like `${diagram('assets/context.png')}` or
  `${image('assets/component.svg')}` depending on how you resolve images. Keep the path relative to the repository root
  or the template’s folder, and resolve it in your resolver.

== Agile/craftsmanship/docs‑as‑code lens

- Shift left the maintenance of visuals. If a sequence flow changes, update the `.puml` next to the test it documents.
  The diagram is re‑generated by CI and stamped into the `.docx`. No more hand‑editing in Word.
- Treat diagrams as testable artifacts. You can write smoke tests that assert the file exists and is non‑empty, or even
  hash the image to detect unexpected changes.
- Prefer text sources over binary. Reviewers can read diffs, suggest changes inline, and link commits to decisions. This
  is documentation‑as‑code applied to visuals.

== Solo maintainer + enterprise usage angle

As the only maintainer, every manual step I remove pays compounding dividends. DaaC cuts down on support questions about
“the diagram doesn’t match the text,” because the same PR updates both. For enterprise teams, it lowers friction with
platform and compliance groups: the diagram toolchain is explicit, repeatable, and reviewable. If a vendor mandates a
specific CI environment, diagrams still build predictably.

== Risks and mitigations

- Risk: tool compatibility in locked‑down environments. Mitigation: prefer Kroki (HTTP) or pre‑approved CLI tools;
  vendor a container image if needed; document the minimal requirements.
- Risk: performance in large repos. Mitigation: generate only diagrams that changed (hash sources); cache results; run in
  a dedicated build step.
- Risk: visual drift if styles are inconsistent. Mitigation: centralize themes (PlantUML skinparams, Mermaid themes) and
  include them from a single source.

== How to apply this

- Pick one diagram type you create often (context, component, sequence). Script its generation and wire it into your
  stamping flow. Keep it minimal and reproducible.
- Add a section in your repo’s docs (“Diagrams”) with one example per tool you support; link to it from your templates.
- Encourage contributions as text. If someone sends a binary `.docx` with a changed picture, ask for the accompanying
  `.puml`/`.mmd` source. That’s how you keep the system teachable.
